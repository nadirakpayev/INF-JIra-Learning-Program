# 04. Subqueries

## Зачем нужен этот модуль

Этот модуль нужен, чтобы участник понимал, как один SQL-запрос может использовать результат другого SQL-запроса.

Подзапросы особенно важны там, где:
- условие фильтрации зависит от других данных;
- нужно сначала вычислить промежуточный результат;
- не хочется писать несколько отдельных запросов руками.

Во всех SQL-примерах ниже используется общий учебный датасет из [00_Mini_Dataset.md](00_Mini_Dataset.md).

## Что нужно понять

### 1. Что такое подзапрос

Подзапрос — это SQL-запрос, вложенный в другой SQL-запрос.

Его результат используется:
- как условие;
- как источник значений;
- как временный набор строк.

Простой пример:
Сначала вы спрашиваете: "какие пользователи набрали больше 10 000 подписчиков?", а потом уже вторым вопросом: "покажи все их посты". Внутренний вопрос и есть подзапрос.

Подробнее:
- [SQL Academy, Подзапросы](https://sql-academy.org/ru/guide/nested-sql-queries)

SQL-пример:
```sql
SELECT title
FROM posts
WHERE author_id IN (
  SELECT id
  FROM users
  WHERE followers_count > 10000
);
```

Сначала выполняется внутренний запрос, потом его результат использует внешний.

### 2. Подзапрос в WHERE

Один из самых частых вариантов — использовать подзапрос внутри `WHERE`.

Это помогает:
- отфильтровать строки по данным из другой таблицы;
- выбирать записи по вычисляемому условию;
- строить более гибкие критерии выборки.

Простой пример:
Сначала мы находим `id` автора самого популярного поста, а потом показываем все остальные посты этого автора.

SQL-пример:
```sql
SELECT title
FROM posts
WHERE author_id = (
  SELECT author_id
  FROM posts
  ORDER BY likes_count DESC
  LIMIT 1
);
```

### 3. Подзапрос как источник данных

Иногда подзапрос используется как временная таблица в `FROM`.

Это полезно, когда:
- нужно сначала подготовить промежуточную выборку;
- после этого уже выполнять дополнительную фильтрацию или соединение;
- важно логически разделить шаги запроса.

Простой пример:
Можно сначала собрать маленький "временный список популярных авторов", а потом уже работать только с ним как с отдельным набором данных.

SQL-пример:
```sql
SELECT popular_users.username
FROM (
  SELECT id, username
  FROM users
  WHERE followers_count > 10000
) AS popular_users;
```

### 4. IN, EXISTS и скалярные подзапросы

На базовом уровне полезно различать:
- `IN` — когда нужно проверить вхождение в набор значений;
- `EXISTS` — когда важно само существование строк;
- скалярный подзапрос — когда подзапрос должен вернуть одно значение.

Простой пример:
`IN` похож на проверку: "автор этого поста входит в список популярных пользователей?". А скалярный подзапрос похож на вопрос: "какой ровно `id` у пользователя с этим логином?"

Подробнее:
- [SQL Academy, Подзапросы с несколькими строками и одним столбцом](https://sql-academy.org/ru/guide/subquery-with-one-column-several-row)
- [SQL Academy, Коррелированные подзапросы](https://sql-academy.org/ru/guide/correlated-subqueries)

SQL-пример с `IN`:
```sql
SELECT title
FROM posts
WHERE author_id IN (
  SELECT id
  FROM users
  WHERE city = 'Almaty'
);
```

SQL-пример со скалярным подзапросом:
```sql
SELECT title
FROM posts
WHERE author_id = (
  SELECT id
  FROM users
  WHERE username = 'alex'
);
```

### 5. Когда подзапрос полезен, а когда мешает

Подзапрос полезен, если:
- он делает запрос понятнее;
- без него логика стала бы слишком разорванной;
- нужно явно выделить промежуточный шаг.

Подзапрос мешает, если:
- его можно заменить более простым `JOIN`;
- он скрывает логику и ухудшает читаемость;
- автор запроса сам не понимает, какой набор строк он возвращает.

Простой пример:
Если вы сами уже запутались, какой список пользователей или постов строится внутри и как он влияет на внешний запрос, значит подзапрос сделан слишком сложно.

Мини-признак хорошего подзапроса:
Сможете ли вы одним предложением объяснить, что возвращает внутренний запрос? Если нет, конструкцию почти наверняка стоит упростить.

### 6. Как читать подзапрос

Читать подзапрос полезно изнутри наружу:
1. Что возвращает внутренний запрос?
2. Сколько строк и каких значений он должен вернуть?
3. Как внешний запрос использует этот результат?
4. Что произойдет, если подзапрос вернет пустой набор?

Мини-пример для разбора:
```sql
SELECT username
FROM users
WHERE id IN (
  SELECT author_id
  FROM posts
  WHERE likes_count > 500
);
```

Сначала внутренний запрос находит авторов популярных постов, потом внешний показывает логины этих пользователей.

## Что нужно уметь после модуля

После этого модуля участник должен уметь:
- дать определение подзапроса;
- объяснить разницу между обычным запросом и подзапросом;
- читать подзапрос в `WHERE`;
- понимать, когда подзапрос возвращает одно значение, а когда набор значений;
- отличать разумное использование подзапроса от избыточного.

## Самопроверка

Проверьте, можете ли вы:
- объяснить, чем подзапрос отличается от обычного запроса;
- прочитать подзапрос сначала отдельно, а потом в составе внешнего запроса;
- объяснить разницу между `IN` и скалярным подзапросом;
- сказать, в каком случае подзапрос проще читать, чем альтернативный `JOIN`.
